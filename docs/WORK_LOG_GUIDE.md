# 작업 일지 사용 가이드

## 개요

작업 일지 시스템은 프로젝트의 일일 작업 내용을 자동으로 정리하고 관리하는 도구입니다. Git 커밋 로그를 기반으로 자동으로 작업 내용을 추출하여 마크다운 파일로 생성합니다.

## 빠른 시작

### 1. 작업 일지 생성

```bash
# 오늘 날짜의 작업 일지 생성/업데이트
npm run work-log

# 특정 날짜의 작업 일지 생성/업데이트
npm run work-log -- 2025-01-15
```

### 2. 작업 일지 확인

생성된 작업 일지는 `docs/work-logs/YYYY-MM-DD.md` 형식으로 저장됩니다.

예: `docs/work-logs/2025-01-15.md`

## 작업 일지 구조

각 작업 일지는 다음 섹션으로 구성됩니다:

### 📋 작업 개요
오늘의 주요 작업 목표와 계획을 작성합니다.

```markdown
## 📋 작업 개요

- [ ] 사용자 인증 기능 구현
- [ ] API 연동 완료
- [ ] UI 컴포넌트 개선
```

### ✅ 완료된 작업
Git 커밋 로그를 기반으로 자동 생성되며, 수동으로 추가할 수도 있습니다.

자동 생성 예시:
```markdown
## ✅ 완료된 작업

### feat: 사용자 인증 기능 추가
- **커밋**: `a1b2c3d`
- **작성자**: 개발자
```
(작업 일지는 날짜 단위만 기록하며, 시간은 포함하지 않습니다.)

### 📝 변경된 파일
Git 로그를 기반으로 자동 생성됩니다.

```markdown
## 📝 변경된 파일

- `src/components/Auth/LoginForm.tsx`
- `src/utils/auth.ts`
- `src/types/user.ts`
```

### 🚧 진행 중인 작업
아직 완료되지 않은 작업을 작성합니다.

```markdown
## 🚧 진행 중인 작업

- 사용자 프로필 페이지 구현 중 (약 60% 완료)
- API 에러 처리 개선 필요
```

### 📅 다음 작업 계획
다음에 할 작업 목록을 작성합니다.

```markdown
## 📅 다음 작업 계획

- [ ] 사용자 프로필 페이지 완성
- [ ] 테스트 코드 작성
- [ ] 문서 업데이트
```

### 💡 이슈 및 메모
발생한 문제나 특이사항을 기록합니다.

```markdown
## 💡 이슈 및 메모

- API 응답 지연 문제 발견 (조사 필요)
- 새로운 라이브러리 도입 검토 중
```

### 📆 작업일
작업 일지는 **날짜 단위만** 기록합니다. 시간(시·분)은 기록하지 않습니다.

```markdown
## 📆 작업일

- **작업일**: 2025-01-15
```

## 자동화 기능

### Git 커밋 기반 자동 생성

작업 일지 스크립트는 다음 정보를 자동으로 수집합니다:

1. **커밋 메시지**: 해당 날짜의 모든 Git 커밋 메시지
2. **커밋 해시**: 각 커밋의 해시 값
3. **작성자**: 커밋 작성자
4. **변경된 파일**: 커밋에서 변경된 파일 목록

(작업 일지에서는 날짜까지만 사용하며, 커밋 시각 등 시간 단위는 기록하지 않습니다.)

### 기존 내용 보존

- 기존 작업 일지 파일이 있으면 내용을 보존합니다
- 새로운 커밋 정보만 추가됩니다
- 수동으로 작성한 내용은 그대로 유지됩니다

## 사용 시나리오

### 시나리오 1: 하루 작업 마무리 시

```bash
# 1. 작업 일지 생성
npm run work-log

# 2. 생성된 파일 열어서 수동 내용 추가
# docs/work-logs/2025-01-15.md

# 3. Git에 커밋
git add docs/work-logs/2025-01-15.md
git commit -m "docs: 작업 일지 추가"
```

### 시나리오 2: 특정 날짜 작업 정리

```bash
# 과거 날짜의 작업 일지 생성
npm run work-log -- 2025-01-10

# 생성된 파일 확인 및 수정
# docs/work-logs/2025-01-10.md
```

### 시나리오 3: 주간 작업 정리

```bash
# 일주일치 작업 일지 생성
npm run work-log -- 2025-01-15
npm run work-log -- 2025-01-16
npm run work-log -- 2025-01-17
npm run work-log -- 2025-01-18
npm run work-log -- 2025-01-19
```

## 팁

### 1. 커밋 메시지 작성 팁

의미 있는 커밋 메시지를 작성하면 작업 일지가 더 유용해집니다:

```bash
# ✅ 좋은 예
git commit -m "feat: 사용자 로그인 기능 구현"
git commit -m "fix: API 에러 처리 개선"
git commit -m "refactor: 컴포넌트 구조 개선"

# ❌ 나쁜 예
git commit -m "수정"
git commit -m "업데이트"
```

### 2. 작업 일지 커밋

작업 일지 파일도 Git에 커밋하여 버전 관리하는 것을 권장합니다:

```bash
git add docs/work-logs/
git commit -m "docs: 작업 일지 업데이트"
```

### 3. 정기적인 업데이트

하루 작업이 끝날 때마다 작업 일지를 업데이트하는 습관을 기르면 좋습니다.

## 문제 해결

### Git 저장소가 아닌 경우

Git 저장소가 아니거나 커밋이 없는 경우, 커밋 정보는 비어있지만 템플릿은 생성됩니다. 수동으로 작업 내용을 작성할 수 있습니다.

### 날짜 형식 오류

날짜는 반드시 `YYYY-MM-DD` 형식을 사용해야 합니다:

```bash
# ✅ 올바른 형식
npm run work-log -- 2025-01-15

# ❌ 잘못된 형식
npm run work-log -- 2025/01/15
npm run work-log -- 01-15-2025
```

## 참고

- 작업 일지 파일은 `docs/work-logs/` 디렉토리에 저장됩니다
- 파일명 형식: `YYYY-MM-DD.md`
- 작업 일지는 마크다운 형식으로 작성됩니다
- 필요시 수동으로 내용을 수정할 수 있습니다
